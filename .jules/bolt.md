## 2024-05-24 - Database Aggregation over Memory Summarization
**Learning:** The `/dashboard` route previously fetched the entire `gym_classes` table into application memory to compute basic statistics like enrollments and class counts, which represents an $O(N)$ memory and payload bottleneck scaling with gym history. Using Drizzle ORM to perform the `sum()`, `count()`, and `groupBy()` logic directly in the database dramatically reduces data transferred over the wire and eliminates the risk of application memory bloat. Additionally, when using Drizzle mock frameworks (like Node's native module mocking), adding database aggregate operators (`lte`, `desc`, `sum`, `count`, etc.) is required to ensure tests run without error.
**Action:** When implementing dashboards or statistics logic, always prioritize database-level aggregations over application-level loops or `reduce()`, especially when filtering large dataset histories. Always coerce sums to `Number` as driver behavior varies between returning numbers or strings.
